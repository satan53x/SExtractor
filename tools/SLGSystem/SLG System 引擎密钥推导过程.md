# 天極姫 SLG System 引擎密钥推导过程

## 背景

天極姫1和2的脚本封包 `script.szs` 使用 SLG System 引擎。  
工具 [SLGSystemScriptTool](https://github.com/TesterTesterov/SLGSystemScriptTool) 内置了一批已知游戏的密钥，但天極姫1和2不在列表中，需要手动推导。

---

## 第一步：解析 SZS 封包格式

`script.szs` 文件头为明文：

```
SZS100__  00 00 00 00  0B 00 00 00  ...
签名(8B)   version(4B)  count=11(4B)
```

索引区每条目 **272 字节**（256B 名称 + 8B offset + 8B size），共 11 个子文件：

| 文件名 | 大小 |
|---|---|
| main.sd | 8,625,634 B（主脚本） |
| main.ev | 261,120 B（事件表） |
| main.lbn | 85,760 B（标签名） |
| ... | ... |

索引区是**明文**，数据区每字节 **XOR 0x90** 加密。解开后数据仍是乱码，说明还有第二层加密。

---

## 第二步：确认第二层加密算法

SLGSystemScriptTool 的 `SLG_Crypto.py` 给出了加密算法：

```python
def _next_key(key):
    key = (key * 0x343FD + 0x269EC3) & 0xFFFFFFFF
    key >>= 16
    key &= 0x7FFF
    return key

# 加解密（XOR 对称）
for each byte:
    key = _next_key(key)
    out_byte = in_byte ^ (key & 0xFF)
```

每个子文件独立从初始密钥开始，密钥流只依赖初始密钥（`init_key`）。  
天極姫不在已知列表，需要暴力或数学推导找 `init_key`。

---

## 第三步：已知明文攻击

### 关键观察

对 XOR 0x90 解包后的文件，偏移 **16\~32** 处的内容：

| 文件 | 字节（hex） |
|---|---|
| main.ev | `cf f7 26 1c fb 3f 95 9a f9 f9 c1 05 99 5f 72 ef` |
| main.lbn | `cf f7 26 1c fb 3f 95 9a f9 f9 c1 05 99 5f 72 ef` |
| main.sfn | `cf f7 26 1c fb 3f 95 9a f9 f9 c1 05 99 5f 72 ef` |

三个不同文件，偏移 16 起完全一样。由于加密是 XOR，密文相同意味着**明文相同**。

这三种文件格式（事件表、标签名表、效果名表）的条目结构都是：
```
[64 字节 名称，后补 \x00] [4 字节 offset]
```
第一个条目的名称通常很短（如 `m_batyou01`，10 字节），其余 54 字节均为 `\x00` 填充。  
因此偏移 10\~64 范围内明文为 **全零**，偏移 16\~32 也是全零。

既然明文为 0x00，则：
```
密文[i] = 0x00 XOR key_stream[i] = key_stream[i]
```

**密文本身就是密钥流**

偏移 16\~32 对应密钥流的第 17\~32 字节（文件从第 1 字节开始消耗密钥流）。

---

## 第四步：数学反推 init_key

已知密钥流第 17\~21 字节为 `cf f7 26 1c fb`，即：

```
_next_key 第 17 次输出 & 0xFF == 0xcf
_next_key 第 18 次输出 & 0xFF == 0xf7
...
```

`_next_key` 输出范围 `0x0000 ~ 0x7FFF`（15 位），所以枚举所有满足 `val & 0xFF == 0xcf` 的值只有 **128 个候选**。

对每个候选验证后续 4 步是否也匹配，最终得到唯一的 `k17`（第 17 步的密钥状态）。

再由 `k17` 反推 `init_key`：

```
k_next = ((k_prev * A + B) & M) >> 16 & 0x7FFF
其中 A=0x343FD, B=0x269EC3, M=0x100000000

反推：k_prev = ((k_next << 16 | low16) - B) * A_inv % M
```

用模逆元 `A_inv = modinv(0x343FD, 0x100000000)` 枚举 `low16`（0\~65535），会得到 65536 个候选 `init_key`。

**注意：这 65536 个候选的解密结果完全相同，无需进一步筛选。** 原因是 `_next_key` 只取第 16\~30 位（`>> 16 & 0x7FFF`），`init_key` 的低 16 位对密钥流没有任何影响，取任意一个候选使用即可。

---

## 结果

| 游戏 | init_key |
|---|---|
| 天極姫 ～新世大乱・双界の覇者達～ | `0x1291f641` |
| 天極姫2~覇権争奪, 幻妖の将星~ | `0xe41ef641` |

验证：用各自密钥解密 main.sd 后，可读出正确的脚本内容，包括文件路径和日文对话文本：

```
Data\bg\effect\shake_flame.tig
平原を騎馬が駆けていた。
「はっ！」
「そろそろ戻るか」
```

## 附：同系列游戏密钥规律

对比两作密钥：

```
天極姫1：0x1291f641
天極姫2：0xe41ef641
```

低 16 位 `0xf641` 完全相同。推测同一编译环境或构建脚本会保留相同的低 16 位，而高 16 位随版本变化。

实际意义：对同系列未知游戏，可以优先尝试固定低 16 位、只暴力高 16 位（共 65536 次，毫秒级），大幅缩短搜索时间。不过由于低 16 位不影响密钥流（见第四步），这更像是构建特征，而非安全设计。

---

## 总结

| 步骤 | 方法 |
|---|---|
| 第一层 | SZS 封包，数据区 XOR 0x90 |
| 第二层 | SLG_Crypto，线性同余密钥流 XOR |
| 密钥发现 | 已知明文攻击（条目名称后的零填充区域） |

`init_key` 对该游戏所有子文件通用，加解密算法完全对称。
